<objective>
Ship the kill switch end-to-end: persistent global halt (`users.kill_active` write path), three surfaces (Slack `/gekko kill CONFIRM`, CLI `gekko kill`, dashboard `POST /kill` typed-KILL form), boot-time persistence check, best-effort parallel cancel-orders with 5s SLA, and the visual + DM affordances per UI-SPEC §2. The READ-side check (`check_kill_switch`) already lives in OrderGuard from plan 02-02; plan 02-05 adds the WRITE side + persistence + UI surfaces.

Per D-37 ordering invariant: set `kill_active=True` FIRST → fetch open orders → parallel cancel with 4s timeout → tally + write `kill_switch` event payload → Slack DM. The READ-side OrderGuard check is the watchdog.

Per D-36 boot-time invariant: FastAPI lifespan SELECTs `users.kill_active`; if True, refuses new orders + DMs the operator + dashboard shows persistent red banner. Resume requires explicit `/gekko unkill UNKILL` with typed confirmation.

Output: 1 new orchestrator module + 2 new Brokerage methods + 2 new templates + 2 new tailwind classes + 3 new routes + Slack slash subcommands + CLI subcommands + the manual-only demo checkpoint.
</objective>
