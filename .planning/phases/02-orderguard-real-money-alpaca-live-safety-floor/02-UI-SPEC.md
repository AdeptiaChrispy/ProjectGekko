# Phase 2 — UI Design Contract

> Visual and interaction contract for the OrderGuard + Real-Money Alpaca Live (Safety Floor) phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

This phase introduces the SEMANTIC RED COLOR FAMILY (live mode, kill switch, OrderGuard rejections), the SEMANTIC YELLOW family (PDT/T+1/wash-sale cautions), and three new surfaces (live banner, kill button + modal, first-live confirm page) on top of the Phase 1 Jinja2 + HTMX + hand-crafted Tailwind subset. The UI's load-bearing job is to make "you are about to commit real dollars" **impossible to miss** and the kill switch **reachable in one click**.

## Design System

| Property | Value |
|----------|-------|
| Tool | none (no shadcn — Python stack, FastAPI + Jinja2 + HTMX per CLAUDE.md; React/Vite gate not applicable) |

**Exceptions for Phase 2:**

- Kill button minimum tap target: 44×44px (accessibility floor; satisfied by 12px vertical × 16px horizontal padding around 16px text label).
- 5-second countdown progress bar height: 4px (smallest scale value — visual heartbeat, not a spacing element).

## Typography

## Surfaces & Interaction Contracts

Five surfaces in scope. Each has BOTH a dashboard form AND (where applicable) a Slack Block Kit parallel.

### Surface 1: `[LIVE — REAL MONEY]` Persistent Red Banner

**Dashboard placement**
- Inserted at the very top of `base.html.j2` body, ABOVE the existing header bar.
- Replaces (when live) — does NOT stack alongside — the existing `<div class="banner-paper">PAPER</div>`. When any live-eligible strategy exists for the operator, the banner is `banner-live-strong`; otherwise the existing `banner-paper` stays.
- Sticky at `top: 0`, `z-index: 50`, full width, 40px tall, NOT dismissible (no close button).
- Banner copy: `[LIVE — REAL MONEY] Alpaca live trading is armed.` (the specific phrasing "REAL MONEY" is mandatory per orchestrator spec.)

**Trigger logic (server-side, evaluated at every page render)**
- Query the per-user SQLCipher DB: `SELECT COUNT(*) FROM strategy_metadata WHERE live_mode_eligible = 1`. Cached on `request.app.state` with 60s TTL to avoid DB-on-every-render.
- If count > 0 → render `banner-live-strong`. Else render `banner-paper`.
- Banner state is a `request.state.banner_mode` value (`"PAPER"` | `"LIVE"`) injected by a FastAPI dependency; all templates inherit.

**State transitions**
- Banner has no client-side state. Page reload re-evaluates. If the operator just demoted (un-promoted) the last live strategy, a full page reload (any link click) refreshes the banner.
- An optional 60s `<meta http-equiv="refresh">` polling is REJECTED for Phase 2 (no auto-refresh; the operator triggers refresh via interaction or explicit reload).

**ARIA / screen-reader**
- Banner is wrapped in `<div role="alert" aria-live="polite" aria-atomic="true">`. The text content "LIVE REAL MONEY Alpaca live trading is armed" reads on every page render. NOT `aria-live="assertive"` — that's reserved for the kill-active banner (Surface 2).
- Screen readers announce the banner on every page load. This is intentional per "impossible to miss" — visual-only signaling is not safety-grade.

**HTMX swap behavior**
- The banner itself does not use HTMX swaps. It is rendered server-side per request. Promotion/demotion that would change banner state requires a full page navigation.

**Slack parallel**
- Every Slack Block Kit card for a `LIVE` proposal carries a header block:
  ```python
  {"type": "header", "text": {"type": "plain_text", "text": "🔴 LIVE — REAL MONEY"}}
  ```
- Below the header, a context block:
  ```python
  {"type": "context", "elements": [{"type": "mrkdwn", "text": "*Alpaca live account.* This will use real capital."}]}
  ```
- The text `LIVE — REAL MONEY` and `Alpaca live account` are deterministic constants — `_escape_mrkdwn` not strictly required (no LLM bytes), but they MUST be routed through the same render path that wraps LLM-authored content. Convention: `slack.render._safe_text("LIVE — REAL MONEY")` returns the literal unchanged for constants and `_escape_mrkdwn`-wraps any LLM-sourced string. One render path = one bug surface.
- PAPER cards stay green ✅ — existing `build_proposal_card(account_mode="PAPER")` path unchanged.

### Surface 2: Kill Switch UI (Navbar Button + Modal + Active Banner)

**Three sub-surfaces in this one feature:**

#### 2a. Kill button (navbar — always visible)

- Position: top-right corner of the `<header>` bar in `base.html.j2`, right of the "Strategies" nav link.
- Label: `KILL` (uppercase, single word, 14px, weight 600, white text on `#dc2626`).
- Always rendered. Whether the operator has live strategies or not — the kill button is the global emergency stop and must be reachable at any time.
- Tap target: minimum 44×44px (accessibility floor); achieved via 12px vertical padding + 16px horizontal padding around the 14px text.
- HTMX attributes:
  ```html
  <button class="btn-kill"
          hx-get="/kill/confirm-modal"
          hx-target="#modal-mount"
          hx-swap="innerHTML"
          aria-label="Open kill switch confirmation. Stops all trading.">
    KILL
  </button>
  ```
- Modal mount: a single `<div id="modal-mount"></div>` lives at the bottom of `base.html.j2`. When empty, no modal. When populated, the modal backdrop overlays the page.

**State of the kill button when `kill_active=true`:**
- Re-labeled to `UNKILL` (still on the danger-red `--gk-kill-btn` background).
- `hx-get` target becomes `/unkill/confirm-modal`.
- `aria-label` becomes "Open unkill confirmation. Resumes all trading."
- Server-side check on every render: `if user.kill_active: render unkill variant else render kill variant`.

#### 2b. Kill confirmation modal (HTMX-loaded fragment)

- Triggered by `hx-get /kill/confirm-modal` from the navbar button.
- Modal copy:
  ```
  Headline: Halt all trading
  Body: This cancels all open orders across all strategies and halts
        every paper and live strategy. The kill state persists across
        process restarts. To resume, you must explicitly UNKILL.

        Type KILL below to confirm.
  Input: <input type="text" name="confirm" required pattern="^KILL$">
  Submit: button "Confirm — halt all trading"
  Cancel: button "Cancel" (closes modal — hx-on="click: document.getElementById('modal-mount').innerHTML = ''")
  ```
  Note: `hx-on` is acceptable under CSP `script-src 'self'` because htmx is loaded from `/static/` — htmx's `hx-on` attribute is interpreted by the local htmx.min.js, not the page-level script context.
  
  Alternative (safer per CSP audit): use `hx-get="/modal/close"` returning empty HTML to clear `#modal-mount`. Preferred for Phase 2.

- Form `hx-post="/kill" hx-target="#kill-banner-mount" hx-swap="outerHTML"`.
- Submit button starts DISABLED. Enabled only when the input value === `"KILL"` (exact match, case-sensitive). Implementation: server-side validates on submit; client-side, htmx 2.x supports `hx-disable` based on form value via the `hx-trigger` modifier — fallback is to omit client-side disable and rely on server-side 400 + re-render with error.
  
  **Recommended Phase 2 implementation:** keep submit always enabled; rely on the server `if confirm.strip().upper() != "KILL": raise HTTPException(400)` already specified in RESEARCH.md §3. The user gets clear feedback either way; defense-in-depth is server-side.

#### 2c. Kill-active banner (appears when kill_active=true)

- Appears BELOW the `banner-live-strong` (or `banner-paper`) — stacked: live banner z-index 50, kill banner z-index 49, sticky `top: 40px` so it sits flush under the live banner.
- 40px tall, `--gk-kill-bg` (`#7f1d1d`) — slightly darker than the live banner so when both are active, the eye registers two distinct safety bars.
- Copy:
  ```
  🚫 KILL ACTIVE — no orders will fire. Cancelled {n_cancelled}/{n_total} open orders.
  ```
- A `<button>` "Unkill" inside the banner (right-aligned, white outline on red bg) opens the unkill confirmation modal (parallel to 2b but typing `UNKILL`).
- ARIA: wrapped in `<div role="alert" aria-live="assertive" aria-atomic="true">`. `assertive` because kill-active is the highest-stakes state — interrupts screen-reader queue.

**Idempotency / double-click defense (Kill flow)**
- Server-side: `_execute_kill(user_id, source)` checks `if user.kill_active: return existing_tally` — second click is a no-op that returns the cached tally.
- The modal closes itself on successful submission (server returns the new `kill-banner-mount` HTML with `hx-swap-oob="true"` on a closing element targeted at `#modal-mount`).
- HTMX `hx-disable-elt="this"` on the submit button prevents in-flight double-clicks; once the response lands, the modal is replaced anyway.

**Loading affordance during the 5s SLA window**
- While the kill is in flight (the executor is cancelling open orders in parallel), the modal swap shows a loading state:
  ```html
  <div class="modal" aria-busy="true">
    <div class="modal-headline">Cancelling open orders…</div>
    <div class="spinner" role="status" aria-label="In progress"></div>
    <div class="modal-body">
      <span hx-get="/kill/state" hx-trigger="every 1s" hx-target="this" hx-swap="innerHTML">
        Setting kill_active=true…
      </span>
    </div>
  </div>
  ```
- The `/kill/state` endpoint polls the in-flight tally and returns text like `"Cancelled 2/3 — 1 pending."` until the kill completes; then the next swap replaces the whole modal with success state + closes after 1.5s.

**Slack parallel (kill flow)**
- Slash command flow (two-step, per RESEARCH.md §3):
  ```
  Operator: /gekko kill
  Bot DM:   ⚠️ Type `/gekko kill CONFIRM` within 60 seconds to halt all trading.
            Currently active strategies: {list of strategy_name (mode)}
            Open orders: {n} across {m} strategies.
  Operator: /gekko kill CONFIRM
  Bot DM:   🚫 Kill ACTIVE. Cancelled {x}/{y} open orders. {z} pending. {w} failed.
            Type `/gekko unkill CONFIRM` when ready to resume.
  ```
- The `CONFIRM` argument is the second-step typed confirmation — symmetric with the dashboard's typed `KILL` and `UNKILL`.
- The bot DM's body MUST route every interpolated value through `_escape_mrkdwn` — even though `strategy_name` is user-authored (not LLM-authored), the same render path applies. Constants like "Kill ACTIVE" pass through unwrapped.

### Surface 3: First-Live Dual-Channel Confirmation (HITL-06)

#### 3a. Slack DM (first channel)

- Existing Phase 1 HITL Block Kit card is REPLACED for first-live trades with a dedicated card variant:
  ```
  Header:  🔴 FIRST LIVE TRADE — DUAL CONFIRM REQUIRED
  Context: *Strategy:* {strategy_name} (live)
           *Ticker:*   {ticker}
           *Action:*   {side} {qty} @ {price}
           *Notional:* ${target_notional_usd}
  Section: (trade rationale summary, mrkdwn-escaped)
  Section: ⚠️ This is your FIRST live trade on this strategy.
           Slack approval alone is NOT enough. Open your dashboard
           to complete the second-channel confirmation.
  Actions: [Open Dashboard to Confirm]  (single button — NO inline Approve/Reject)
  ```
- The single "Open Dashboard to Confirm" button is a Block Kit URL-button:
  ```python
  {
      "type": "actions",
      "elements": [{
          "type": "button",
          "text": {"type": "plain_text", "text": "Open Dashboard to Confirm"},
          "style": "primary",
          "url": f"{settings.dashboard_url}/live-confirm/{proposal_id}",
      }],
  }
  ```
- A URL-button is intentional — it does NOT generate a Slack `action_id` round-trip. The user is forced to leave Slack and land on the dashboard. The Slack channel alone is **not** sufficient to advance the trade state.
- Existing Phase 1 HITL behavior on the regular HITL card (Approve / Reject inline buttons) is **suppressed** for first-live proposals — the state-machine transition `PENDING → APPROVED` is NOT exposed in Slack for first-live trades. The state machine `_approve_workflow` in `src/gekko/approval/slack_handler.py` for first-live proposals diverts to `AWAITING_2ND_CHANNEL` without an Approve button.

  **Refined:** The Slack DM has NO Approve/Reject for first-live — only "Open Dashboard". Rejection happens on the dashboard confirm page (Cancel button) which transitions `AWAITING_2ND_CHANNEL → REJECTED`.

#### 3b. Dashboard `/strategies/{name}/first-live-confirm/{proposal_id}` page (second channel)

- A standalone full-page template (extends `base.html.j2`), NOT a modal. The operator MUST be on this page, viewing the trade details, to confirm.
- Layout (top to bottom):
  1. **Live banner** (Surface 1) at the very top — already inherited from base.
  2. **Page hero** (24px display headline): "Confirm First Live Trade — {strategy_name}"
  3. **Trade detail panel** (48px below hero, white card with neutral border):
     ```
     Ticker:           AAPL
     Action:           BUY 25 shares
     Limit price:      $187.50
     Estimated cost:   $4,687.50  (target_notional_usd)
     Strategy:         {strategy_name} (live)
     Proposal ID:      {proposal_id}
     Rationale:        (full text, not truncated)
     ```
  4. **Trust acknowledgement block** (32px below trade detail, caution-yellow surface — NOT red, because this is a checkbox, not a stop):
     ```
     [ ] I understand this is real money on my Alpaca live account.
     [ ] I have read the rationale and trade details above.
     ```
  5. **Countdown timer** (24px below acknowledgements): "Read time: 5… 4… 3… 2… 1… (you may now confirm)"
     - Implementation: HTMX polling. Server returns the current countdown numeral every 1s for 5s; after 5s, the response replaces the timer area with the words "Read time complete" AND triggers the confirm button to enable via OOB swap.
     - Alternative simpler approach: render the countdown server-side once, page-load timestamp + 5s. The button has `disabled` HTML attribute server-rendered with `disabled` until both checkboxes are ticked AND the 5-second timestamp has passed. The button polls `/live-confirm/{id}/can-confirm` once per second; when the server returns enable=true, the button's `disabled` attribute is removed via OOB.
     - **Phase 2 recommendation: pure server-side timestamp check, no client-side state.** The button stays disabled until BOTH (a) the form has `checkbox_1=on&checkbox_2=on` (server check at POST time) AND (b) `time.time() - page_load_ts >= 5.0` (server check at POST time). The countdown numeral is visual-only; the button is always-clickable but server rejects with HTTP 400 + re-render if either gate fails. Simpler than the OOB dance and equally safe.
  6. **Confirm + Cancel buttons** (64px below acknowledgements):
     - Primary: `<button type="submit">Confirm First Live Trade</button>` — the SAME button color as a normal "Approve" (blue, `#2563eb`), NOT red. Red would imply "destructive" — a live trade confirmation is the opposite, it's the intended action. The red banner above the button carries the warning.
     - Secondary: `<a href="/strategies">Cancel — return to strategies</a>` — a text link, neutral gray, server-side transitions `AWAITING_2ND_CHANNEL → REJECTED`.

- Form HTMX:
  ```html
  <form hx-post="/live-confirm/{proposal_id}"
        hx-target="#confirm-result"
        hx-swap="outerHTML">
    <input type="hidden" name="page_load_ts" value="{{ page_load_ts }}">
    <input type="checkbox" name="ack_real_money" required>
    <input type="checkbox" name="ack_read_rationale" required>
    <button type="submit">Confirm First Live Trade</button>
  </form>
  <div id="confirm-result"></div>
  ```

**State transitions**
- Idle → form filled → submit → server validates (typed checkboxes + 5s elapsed) → transition `AWAITING_2ND_CHANNEL → APPROVED_LIVE` → render success partial → executor dispatched async.

**Idempotency**
- The dashboard route `POST /live-confirm/{proposal_id}` (already specified in RESEARCH.md §7) checks the current row status:
  - `AWAITING_2ND_CHANNEL` → transition to `APPROVED_LIVE`, dispatch executor, return success template.
  - `APPROVED_LIVE` (already confirmed once) → return success template with the additional line "This trade was already confirmed at {timestamp}. Executor dispatched."
  - any other status → 400 "wrong state".
- Server-side idempotency is the load-bearing defense; client-side `hx-disable-elt="this"` on the submit button is additional polish, not the primary mechanism.

**ARIA**
- Page headline `<h1>` with `id="page-headline"`.
- Trade detail panel: `<section aria-labelledby="trade-detail-heading">` with an `<h2 id="trade-detail-heading">Trade Detail</h2>`.
- The checkboxes are standard `<label><input type="checkbox">`; native form semantics, no ARIA overrides needed.
- The countdown text region: `<div aria-live="polite">`. The numerals update every 1s; screen reader announces them politely (not assertive — we want a heads-up, not an interrupt).
- Disabled submit button: native `disabled` attribute (screen readers announce "button, dimmed" automatically).
