<objective>
Land HITL-02 end-to-end: a belt-and-suspenders idempotency gate at the top of every Slack action handler (`handle_approve` + `handle_reject`) that catches Slack's at-least-once retries via the `slack_action_dedup` UNIQUE constraint AND uses the existing state-machine CHECK as the secondary defense. Duplicates respond with an ephemeral D-43 toast naming the original first-writer; first-writes proceed to the existing state-transition path.

Purpose: HITL-02 is the single most-cited HITL pitfall across all four researchers (Slack's at-least-once delivery is documented and not negotiable). Without this plan, a slow operator click + a Slack retry = two executions of the same proposal. The dedup table from Plan 03-01 is the atomic primitive; this plan is the handler-side glue.

Output:
- `src/gekko/approval/dedup.py` NEW module with `claim_action(...)` helper (the load-bearing INSERT-with-IntegrityError-catch shape mirrored from `agent/proposal_writer.py:290-329`).
- `src/gekko/approval/slack_handler.py` EXTENDED: `_approve_workflow` and `_reject_workflow` insert a dedup row immediately after the cross-user check (lines 141-161 substrate); on `duplicate`, fire ephemeral via `respond_url` (D-43) and return without touching state. `body.get("trigger_id")` is plumbed through to `claim_action(slack_trigger_id=...)` for retry-header debugging per D-45.
- `tests/unit/test_slack_action_dedup.py` filled: first-write returns 'first_write'; second identical INSERT returns 'duplicate' without raising; dedup_click audit event lands on the duplicate path.
- `tests/unit/test_slack_retry_header.py` filled: X-Slack-Retry-Num retries gated before the INSERT branch.
- `tests/integration/test_dedup_race.py` filled: cassette-style approve + edit race produces exactly one APPROVED state transition.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Slack HTTP -> Bolt async handler | Slack signs payloads with `SLACK_SIGNING_SECRET`; slack-bolt verifies before our handlers run (Plan 01-08 lock). Trust is established BEFORE `claim_action`. |
| body["user"]["id"] -> handler-side authoritative actor | A spoofed body could in principle carry a wrong user_id, but slack-bolt's signature-verify gate makes this only possible from an attacker holding the signing secret. P3 does NOT widen this attack surface; cross-user check at lines 141-161 is the additional defense. |
| Concurrent INSERT to slack_action_dedup ↔ UNIQUE constraint | SQLCipher / SQLite WAL serializes writers; concurrent INSERTs on the same UNIQUE triplet produce a deterministic winner + a deterministic IntegrityError on the loser. No race. |
| respond_url -> Slack (outbound HTTP) | The `response_url` Slack provides is a single-use URL with ~30min TTL. Posting failed (e.g., expired) is logged but does not raise — the dedup row already exists so the state truth is preserved. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-02-01 | Spoofing | Attacker fires a synthetic interactivity payload claiming to be the operator's slack_user_id | mitigate | slack-bolt signature-verify gate runs BEFORE handler entry (Plan 01-08 lock; `SLACK_SIGNING_SECRET` env var); plus cross-user check at slack_handler.py:141-161 |
| T-03-02-02 | Tampering | Race condition between approve + reject INSERTs corrupting state | mitigate | UNIQUE constraint on `(proposal_id, action_id, actor_slack_user_id)` (Plan 03-01) is the atomic primitive; state-machine `transition_status` is the secondary defense; integration cassette test_dedup_race.py provably enforces exactly-once |
| T-03-02-03 | Repudiation | Operator denies clicking approve | mitigate | Every claim_action INSERT records `actor_slack_user_id`, `actor_gekko_user_id`, `slack_trigger_id`, `inserted_at` per D-45; `dedup_click` event appended on every duplicate per D-45 |
| T-03-02-04 | Information disclosure | Ephemeral message leaks the original first-writer's Slack ID to a different clicker | accept | The ephemeral is sent ONLY to the duplicate-clicker (Slack's `response_type="ephemeral"` semantics); the original first-writer's Slack mention is the load-bearing identity signal — without it the operator cannot tell who actually approved.
