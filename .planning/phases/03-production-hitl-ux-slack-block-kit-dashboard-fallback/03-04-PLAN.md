<objective>
Land HITL-03 end-to-end: a 60-second APScheduler sweep that expires PENDING proposals whose `expires_at` has passed (per D-50 / D-51 / D-61), transitions them to EXPIRED, writes the `expiration` audit event, mutates the original Slack card in-place via `chat.update` to the greyed-out variant (per D-53), and DMs the operator with the expiry notice. Edge case (sweep vs click race) is resolved by the existing state-machine idempotent same-state return (line 139-141 from Plan 03-01 Task 3) plus the dedup table from Plan 03-02.

Purpose: Without timeout=REJECT a sleeping operator waking up to a stack of stale PENDING proposals is the v1 failure mode HITL-03 was scoped to fix. The 60-second sweep granularity matches the v1 swing-horizon strategy cadence (no need for sub-minute precision); the restart-safe APScheduler knobs (`coalesce=True`, `max_instances=1`, `misfire_grace_time=300`) survive `gekko serve` restarts without double-firing or missed sweeps.

Output:
- `src/gekko/approval/expiry.py` NEW module with `expire_stale_proposals(*, user_id)` async sweep.
- `src/gekko/scheduler/jobs.py` EXTENDED: `register_expire_stale_sweep(scheduler, *, user_id)` registrar per PATTERNS §2f.
- `src/gekko/dashboard/app.py` EXTENDED: lifespan registers the sweep alongside the existing P1 jobs after `app.state.scheduler.start()`.
- `src/gekko/reporter/slack.py` EXTENDED: `build_proposal_card(expired=True)` branch swaps the actions block for a context block with the expired-state copy from UI-SPEC §Surface 4.
