<objective>
Replace the unusable 2%-drift-vs-target-notional gate on the edit-size modal with
a legible, cap-based validation that validates the operator's edited quantity against
the strategy's OrderGuard hard caps (max_position_pct * account equity ≈ max order size
in dollars; convert to shares via ref_price). Both the Slack Block Kit modal and the
dashboard HTMX modal share the same server-side helper so validation cannot diverge.

Update 03-UI-SPEC.md Surface 1 (Slack edit-size modal) and D-54 in 03-CONTEXT.md to
reflect the new contract.

Purpose: The 2% drift check is the agent's own output-consistency guard (D-27). It is
correct to apply it when the AGENT produces a quantity. It is wrong to apply it when the
OPERATOR deliberately resizes — that makes edit-size useless (even 47→50 shares, ~6%
change, was rejected). The real Knight Capital defense is the absolute risk bounds:
max_position_pct * account equity (or the configurable max_daily_loss_usd). These bound
any operator resize just as they bound any agent-proposed order.

Output:
- actions.py: new _check_edit_size_caps(qty, ref_price, strategy, broker) helper
- slack_handler.py: handle_edit_size_view_submission uses cap check, not drift
- routes.py: edit_size_submit uses cap check, not drift
- edit_size_modal.html.j2: plain-language framing + live $total cost line
- 03-UI-SPEC.md Surface 1 + 03-CONTEXT.md D-54 updated to new contract
- tests/unit/test_edit_size_caps.py: acceptance tests
</objective>
