<objective>
Triage and fix the "BrokerOrderError: broker not configured; falling back to yahooquery"
failure that blocks paper order placement when the operator approves from the dashboard.

The error message originates in src/gekko/agent/tools/alpaca_data.py line 126 — it is
the get_quote Researcher tool's fallback when ctx.broker is None. That file is NOT in the
executor path. This means the log message that appears during approve is from a DIFFERENT
call site — either a stale log entry being misread, or the executor is printing the same
string via a different route.

First task: triage — read executor.py._build_broker carefully to confirm whether the
paper-path AlpacaBroker is actually being constructed and whether the settings env vars
(alpaca_paper_api_key, alpaca_paper_secret_key) are populated at runtime. The
"broker not configured" string only lives in alpaca_data.py; the executor path should
never emit it.

Hypothesis (most likely): The settings env vars are present (the fill stream connects
successfully at startup), _build_broker constructs the broker correctly, but the
execute_proposal task fails at a DIFFERENT point (market_closed guard fires because the
market is closed during testing) and the proposal goes FAILED — the operator
misattributes the log message from alpaca_data.py (which fires during Researcher agent
runs) to the executor path.

Second hypothesis: routes.py approve_proposal_endpoint dispatches with wrong user_id.

Third hypothesis: real wiring gap in how execute_proposal accesses the strategy row.

Purpose: Unblock the execution half of DASH-04 / SC-5. Approve from dashboard must
produce a real fill, identical to the Slack approval path.

Output: A targeted fix to routes.py or executor.py; a test proving the paper-approve
path works end-to-end (monkeypatched broker); a diagnosis note added to the plan summary.
</objective>
