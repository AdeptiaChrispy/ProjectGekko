# Phase 3 — UI Design Contract

> Visual and interaction contract for the Production HITL UX (Slack Block Kit + Dashboard Fallback) phase. Generated by gsd-ui-researcher.

This phase ships **seven new surfaces** on top of the Phase 1 Jinja2 + HTMX + hand-crafted Tailwind subset and the Phase 2 safety semantic tokens (red family for live/kill, yellow for caution). The UI's load-bearing job is to make every approval surface — Slack and dashboard — feel like a single coherent product, so the operator can complete approve/reject/edit-size from either transport, never see double-execution under Slack's at-least-once delivery, never get paged at 2am, and always know whether a card is still actionable or already expired.

**P3 inherits the entire Phase 2 design system verbatim.** P3 does NOT redefine tokens P2 already locked. The Phase 2 UI-SPEC at `.planning/phases/02-orderguard-real-money-alpaca-live-safety-floor/02-UI-SPEC.md` is the load-bearing reference.

## Spacing Scale

Inherits Phase 2's 4-multiple grid verbatim. No new tokens, no new exceptions:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline icon gap, chip border-radius |
| sm | 8px | Compact element spacing (button-to-input gap inside modals, between proposal-card buttons) |
| md | 16px | Default element spacing (card padding, section gaps, login form vertical rhythm) |
| lg | 24px | Section padding (modal body, approvals-index page padding, proposal-card outer margin) |
| xl | 32px | Layout gaps (between header and proposal-card list, between login hero and form) |
| 2xl | 48px | Major section breaks (between approvals-index header and "Empty state" / list region) |
| 3xl | 64px | Page-level spacing (above the login submit button to give the form room to breathe) |

**Policy (P2 carry-forward):** outer layout (gaps between sections, container margins, grid columns) uses 4-multiples only. Sub-component padding may use non-4 values where typographic centering math or WCAG accessibility floors require — but only when the resulting outer dimension lands on a 4-multiple.

**Phase 3 sub-component exceptions (named, bounded):**

- **Login form `<input type="password">` height:** 40px fixed (matches Phase 2 live banner). 10px+10px vertical padding + 14px text × 1.4 line-height = ~40px outer.
- **Approve / Reject / Edit-size / Escalate dashboard buttons:** minimum 44×44px tap target (WCAG 2.5.5 accessibility floor — Phase 2 lock). Achieved via 14px text + 12px vertical / 16px horizontal padding.
- **Slack modal width:** controlled by Slack, not us. Block Kit modal width is fixed by the Slack client renderer (~520px desktop) — we do NOT specify pixel widths inside `views_open` payloads.
- **Expired-card opacity overlay:** 40% (0.4 alpha) — visual-only signal that the card is no longer actionable. NOT a hard CSS variable; declared in the new `.proposal-card.expired` rule.

## Surfaces & Interaction Contracts

Seven surfaces in scope. Each is declared with placement, copy, state transitions, idempotency posture, ARIA semantics, and (where applicable) Slack parallel.

### Surface 1: Slack Edit-Size Modal (D-54) — updated Plan 03-11

**Trigger:** operator clicks the `Edit size` button on a PENDING proposal card in Slack. The button's `action_id="edit_size"` callback fires `views_open` with `private_metadata = proposal_id`.

**Plan 03-11 change:** `_drift_check` is NOT called for operator edits. `_check_edit_size_caps` is the sole gate. See `src/gekko/approval/actions.py`.

**Modal payload shape (slack-bolt `views_open`):**

```python
{
    "type": "modal",
    "callback_id": "edit_size_modal",
    "private_metadata": json.dumps({
        "decision_id": proposal_id,
        "ref_price": str(ref_price),
        "target_notional_usd": str(target_notional),
        "original_qty": str(original_qty),
        "ticker": tp.ticker,
        "side": side_str,   # e.g. "BUY"
    }),
    "title":  {"type": "plain_text", "text": f"Edit order size — {side_str} {original_qty} {tp.ticker} (~${original_notional:,.2f})"},
    "submit": {"type": "plain_text", "text": "Approve at this size"},
    "close":  {"type": "plain_text", "text": "Cancel"},
    "blocks": [
        # Block 1 — qty input
        {
            "type": "input",
            "block_id": "qty_block",
            "label": {"type": "plain_text", "text": "New quantity"},
            "element": {
                "type": "number_input",
                "action_id": "qty_input",
                "is_decimal_allowed": True,
                "initial_value": str(original_qty),
                "min_value": "0",
            },
        },
        # Block 2 — plain-language framing (read-only context block)
        {
            "type": "context",
            "elements": [{
                "type": "mrkdwn",
                "text": (
                    f"Current: {side_str} {original_qty} {tp.ticker} (~${original_notional:,.2f})\n"
                    f"Ref price: ${ref_price}\n"
                    "Adjust shares below — cap enforced at submit"
                ),
            }],
        },
        # Block 3 — validation context
        {
            "type": "context",
            "elements": [{
                "type": "mrkdwn",
                "text": (
                    "Your change is validated against your strategy's risk caps,"
                    " not against the agent's original target."
                ),
            }],
        },
    ],
}
```

**Cap-exceeded error (server-side validation in `handle_edit_size_view_submission`):**

When `_check_edit_size_caps(new_qty, ref_price, strategy, account_equity)` returns `(False, msg)`, respond with:

```python
{
    "response_action": "errors",
    "errors": {"qty_block": "That's above your max of $12,000.00 (~60 shares) — pick a smaller number."}
}
```

The qty input retains the operator's last-typed value. NO state change, NO audit event on a cap failure. The modal stays open; the operator either fixes the qty or cancels.

`_drift_check` is NOT called for operator edits. `_check_edit_size_caps` is the sole gate.

**State transitions (success path):**

1. Operator submits qty within the strategy's hard cap (`max_position_pct * account_equity`).
2. Server INSERTs dedup row with `(proposal_id, action_id='edit_size', actor_slack_user_id, source='slack')` per D-41.
3. Validates via `_check_edit_size_caps(qty, ref_price, strategy, account_equity)` — OrderGuard hard caps, NOT 2% drift.
4. Writes `edit_size` audit event with payload `{old_qty, new_qty, old_notional, new_notional}` per D-45.
5. Updates `proposal.qty`, transitions `PENDING → APPROVED`, dispatches to executor.
6. Closes modal (Slack closes on 200 OK response).
7. The original proposal card in the channel updates via `chat.update` to show `[APPROVED]` chip + final qty.

**Idempotency posture:**

- Submitting the modal twice (double-click within Slack's at-least-once window) hits the dedup table's UNIQUE constraint on `(proposal_id, action_id, actor_slack_user_id)`. `IntegrityError` branches to the D-43 ephemeral path.
- Submitting a modal whose proposal was already approved-via-channel-edit in another transport (dashboard `/approvals/{id}/edit-size`) sees `transition_status(PENDING → APPROVED)` raise `InvalidStateTransition`. The handler catches, returns the D-43 ephemeral via `respond_url`.

**ARIA / accessibility:**

- Slack handles modal a11y semantics natively — Slack client renders an accessible dialog. We do NOT inject custom ARIA inside Block Kit payloads.
- The plain-language cap error ("That's above your max of $X (~N shares)") is explicit text; screen readers do not depend on glyph semantics.

**Slack parallel:** this IS the Slack parallel — there's a dashboard parallel below (Surface 2c, the HTMX modal swap inside `/approvals`).

### Surface 2: Dashboard `/approvals` Index (D-55, DASH-04)

> "The card is scannable at-a-glance: SIDE/QTY/TICKER is the action, $cost is the exposure, the 1-line summary is the why. Full rationale/evidence are secondary — collapsed by default under `<details>`. This layout was prototyped live during UAT 2026-06-19 and formalized here."

### Surface 7: Dedup Ephemeral Slack Message (D-43)

**Trigger:** when `IntegrityError` lands on the dedup-table INSERT inside any of `handle_approve` / `handle_reject` / `handle_edit_size_submit` / `handle_escalate`, the handler immediately responds via Slack's `respond_url` with `response_type="ephemeral"`. Ephemeral = only the clicker sees it; no channel noise (D-43).

**Message shape — exactly one of four variants (deterministic, no LLM bytes):**

| Original action | Current proposal status | Ephemeral copy |
|-----------------|------------------------|----------------|
| Approve (duplicate) | APPROVED / APPROVED_LIVE / AWAITING_2ND_CHANNEL | `✅ Already approved by @{slack_user} at {HH:MM}. Status: {current_status}.` |
| Reject (duplicate) | REJECTED | `❌ Already rejected by @{slack_user} at {HH:MM}.` |
| Edit-size (duplicate) | APPROVED (via prior edit-size submit) | `✏️ Already edited and approved by @{slack_user} at {HH:MM}. New qty: {new_qty}.` |
| Any action on already-expired proposal | EXPIRED | `⏰ Proposal already expired at {HH:MM}.` |

**Copy locked:**

- Approved variant: `✅ Already approved by @{slack_user} at {HH:MM}. Status: {current_status}.`
- Rejected variant: `❌ Already rejected by @{slack_user} at {HH:MM}.`
- Edit-size variant: `✏️ Already edited and approved by @{slack_user} at {HH:MM}. New qty: {new_qty}.`
- Expired variant: `⏰ Proposal already expired at {HH:MM}.`

**Sender resolution:**

- `@{slack_user}` resolves to the `slack_user_id` from the original `slack_action_dedup` row — NOT from the current duplicate click's actor. This is the identity-split-safe path (per `quick/260612-nlv` lesson): the ephemeral message names the person who actually approved (the first writer), not the person who just clicked the duplicate.
- The handler renders the Slack mention syntax `<@U08LRFFRBS4>` so Slack auto-resolves to the display name client-side.

**Timing:**

- Ephemeral messages have a Slack server-side TTL of ~30 minutes after the original trigger (per Slack API docs — `respond_url` lifetime). If the duplicate click lands more than 30min after the original action, the ephemeral fails silently; the operator sees nothing. This is acceptable — by 30min the proposal is also expired (D-51 default 30min timeout), so the operator has visibility via the expired-card update path (Surface 4).

**State transitions:** none. The ephemeral is a UI-only feedback layer; the dedup row + state-machine CHECK are the load-bearing primitives. The ephemeral handler:

1. Catches `IntegrityError` (or `InvalidStateTransition`).
2. Looks up the original dedup row to extract `inserted_at` (for the `HH:MM` value) and `actor_slack_user_id` (for the mention).
3. Looks up `proposals.status` to determine which variant of the ephemeral to send.
4. POSTs to `respond_url` with the variant copy.
5. Writes a `dedup_click` audit event per D-45 schema.

**Idempotency posture:**

- The ephemeral itself is fire-and-forget. A double-ephemeral (e.g., operator triple-clicks) produces multiple ephemerals — acceptable, since each one is private to the clicker and disappears on dismiss.
- The `dedup_click` audit-event write IS idempotent on `(proposal_id, action_id, actor_slack_user_id, inserted_at)` — same UNIQUE constraint family as the dedup table — so triple-clicks produce one audit event, not three.

**ARIA / accessibility:** Slack handles ephemeral message a11y natively. No custom semantics.

**Slack parallel:** this IS the Slack surface. The dashboard parallel is the HTMX swap that updates the stale card with the current truth (Surface 2 idempotency note — "the visual state update IS the feedback").

### Destructive Action Confirmations

P3 introduces NO new destructive actions. All existing destructive flows (kill switch, first-live trade confirmation, promote-to-live) are P2-locked and inherited unchanged. The Reject button on a proposal card is NOT destructive in the P2-locked sense (it doesn't commit real money or halt the system) — it transitions the proposal to REJECTED, which is the safer-side default. Single click, no typed confirmation, matching P2 first-live's "Cancel — return to strategies" link semantics.

**Edit-size IS NOT destructive even though it mutates qty:** the OrderGuard 2% drift check (D-27 / D-54) is the load-bearing safety boundary. The modal CAN be over-approved if drift is within 2% — that's by design (operator chose this qty within the safety envelope). The defense is the drift check, NOT a typed confirmation. This preserves the Knight Capital prevention without adding cognitive friction for routine size tweaks.
