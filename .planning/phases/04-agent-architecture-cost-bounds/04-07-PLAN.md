    upgrade() body — two operations:

    OPERATION 1 — Repair existing corrupted rows. Use a deterministic SQL UPDATE via
    op.execute() targeting the exact 6-character over-quoted value. The stored value is
    literally the 6-char string: apostrophe + 5 + . + 0 + 0 + apostrophe (i.e. "'5.00'").
    The correct repair target is the 4-char clean string "5.00". Write the UPDATE as:

      op.execute(
          "UPDATE users SET daily_cost_ceiling_usd = '5.00' "
          "WHERE daily_cost_ceiling_usd = '''5.00'''"
      )

    That WHERE clause is: SQLite literal for a string containing single quotes requires
    doubling the inner quotes. The value to match is the 6-char string: ' 5 . 0 0 '.
    In SQLite SQL: '''5.00''' means: outer single quotes delimit the string, inner doubled
    single quotes are literal apostrophes — yielding the string '5.00' (with apostrophes).
    This UPDATE is idempotent: on a DB where the value is already clean "5.00" the WHERE
    never matches, so no rows are touched.

    OPERATION 2 — Correct the column server_default going forward so NEW rows after
    migration 0006 get a clean "5.00" without surrounding quote chars. Use batch_alter_table
    to alter the column (batch required for SQLite column changes per project convention):

      with op.batch_alter_table("users") as bop:
          bop.alter_column(
              "daily_cost_ceiling_usd",
              existing_type=sa.String(),
              server_default="5.00",
              nullable=True,
          )

    Note: server_default="5.00" (no surrounding single quotes). SQLAlchemy renders this as
    the DDL DEFAULT '5.00', which SQLite stores as the 4-char clean string. This is in
    contrast to 0005's server_default="'5.00'" which rendered as DEFAULT '''5.00''' and
    stored the 6-char over-quoted string. The pattern file (04-PATTERNS.md §0005) documents
    this bug explicitly.

    downgrade() body — reverse ONLY the column default change (the data repair is
    intentionally NOT reversed — downgrading to 0005 does not re-corrupt the data):

      with op.batch_alter_table("users") as bop:
          bop.alter_column(
              "daily_cost_ceiling_usd",
              existing_type=sa.String(),
              server_default="'5.00'",
              nullable=True,
          )
