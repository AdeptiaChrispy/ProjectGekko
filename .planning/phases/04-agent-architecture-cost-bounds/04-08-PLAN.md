<objective>
Fix the cost-alert dedup regression in `check_cost_ceiling`: the function sets
`user.cost_alert_80_sent_date` / `user.cost_alert_100_sent_date` and then only calls
`await session.flush()` inside an `async with session_factory() as session:` block
opened WITHOUT `.begin()`. An `AsyncSession` rolls back on context-exit with no commit,
discarding the UPDATE every cycle. This causes the 80%/100% Slack DMs to re-fire on
every trigger instead of once per day — violating D-06/D-08/D-12.

Purpose: Restore the "one DM per threshold per day" invariant so the live operator sees
exactly one 80% degrade DM and one 100% halt DM per UTC/tz day, with no spam on
subsequent same-day triggers. This is an alert-spam fix; the halt enforcement path is
correct and unaffected.

Output: Updated `cost_ceiling.py` with a committed sent-date write, plus a real-session
integration test that is guaranteed to fail against the pre-fix code and pass after.
The operator re-tests by lowering the ceiling, triggering two cycles above threshold,
and confirming only ONE DM arrives each time. No migration or schema change required —
restart `gekko serve` to pick up the fix.
</objective>

    The key structural change: replace the single outer context manager
      `async with session_factory() as session:`
    with the two-manager form that opens a transaction:
      `async with session_factory() as session, session.begin():`
    This is the project's established commit idiom (used in proposal_writer.py:349 and
    the seed helper in test_dashboard_hitl_actions.py:126). When the context manager pair
    exits normally, SQLAlchemy commits; when it exits via exception, it rolls back.

    The session block currently has TWO exit paths that return early before reaching the
    flush (fail-open: user-not-found at line 136). Those early returns are inside the
    `try:` / `finally:` wrapper that disposes the engine — they are NOT inside the session
    context manager itself. The early `return CeilingCheck(action="allow", ...)` at
    line 139 exits the `async with session_factory() as session:` block cleanly, which
    with `.begin()` will commit an empty transaction (no-op, safe). No change to the
    fail-open logic is needed.

    The read-only path (pct < 80%, no threshold crossed): `just_crossed_80` and
    `just_crossed_100` are both False, so the `if just_crossed_80 or just_crossed_100:`
    guard at line 239 is not entered. With `.begin()`, the transaction still commits at
    exit, but it is a read-only transaction with no staged mutations — commit is a no-op
    (safe, no DB write occurs).

    CHANGE SUMMARY (surgical, single-line diff):
    - Line 132: `async with session_factory() as session:`
      becomes: `async with session_factory() as session, session.begin():`
    - Remove line 239-240 (`if just_crossed_80 or just_crossed_100: await session.flush()`)
      because `.begin()` commits all staged ORM mutations on exit — the explicit flush is
      no longer the only way to push state. You MAY keep the flush before the commit for
      write-ordering clarity, but it is not required. If kept, keep it; do NOT change it
      to a commit() call (double-commit causes an error with begin()).

    Update the module docstring bullet at lines 20-22 ("One DM per day" / D-06/D-08)
    to note the commit mechanism: `… committed via session.begin() context manager …`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| DB write (sent-date) | `check_cost_ceiling` writes `cost_alert_*_sent_date` to the User row — this is a privileged mutation; it must not be reachable by LLM output (the function is deterministic, LLM-free) |
| Fail-open early return | User-not-found returns `action="allow"` — this early return must not commit an unintended write; with `session.begin()` an empty-mutation commit is a no-op (safe) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-08-01 | Tampering | cost_ceiling.py — sent-date write | mitigate | `session.begin()` wraps the write in an explicit transaction; the ORM cannot silently discard the UPDATE on context-exit; rollback only on exception (correct) |
| T-04-08-02 | Denial of Service | alert spam (the bug being fixed) | mitigate | Commit ensures `cost_alert_*_sent_date` persists, so subsequent calls in the same day return `just_crossed=False` and no DM fires — alert spam eliminated |
| T-04-08-03 | Tampering | fail-open path (user not found) | accept | Early return exits `session.begin()` block with no staged mutations → commit is a no-op; fail-open semantics preserved; low risk (internal gate, no external input) |
| T-04-08-04 | Elevation of Privilege | halt bypass via un-persisted dedup | accept | Halt ENFORCEMENT is separate from alert dedup (runtime.py gates on `_ceiling.action == "halt"`); the bug only affects DM spam, not halt correctness; fix is additive safety |
| T-04-08-SC | Tampering | npm/pip/cargo installs | accept | No new packages introduced in this plan; existing dependencies unchanged |
</threat_model>
