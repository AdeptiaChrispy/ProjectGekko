# Phase 5 — UI Design Contract

> Visual and interaction contract for the Trust Ladder (Per-Strategy Promotion & Portfolio Caps) phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

This phase ships the **autonomy axis** of the trust model on top of the existing paper/live axis (Phase 2) and the cost/quiet-hours settings surfaces (Phase 3/4). Its load-bearing UI job is to make the act of **handing real-money execution authority to an autonomous agent** feel deliberate, reversible, and legible: a promote step that reads like a sit-down decision (never a phone tap), a one-click demote that is always available, a separate capital-scaling rung that doesn't get conflated with autonomy, a blocked-promotion state that *explains the unmet criterion* instead of failing silently, and auto-executed-trade surfacing that keeps the operator informed without re-introducing the HITL gate the promotion was meant to remove.

**P5 inherits the entire Phase 1→4 design system verbatim.** It does NOT redefine any token Phases 1–4 already locked. The canonical references are:

- `.planning/phases/02-orderguard-real-money-alpaca-live-safety-floor/02-UI-SPEC.md` — full color matrix (live/kill red family, caution yellow, neutral gray, accent blue, success green), spacing grid, typography roles, destructive-confirmation pattern (typed-strategy-name).
- `.planning/phases/03-production-hitl-ux-slack-block-kit-dashboard-fallback/03-UI-SPEC.md` — proposal-card partial, chip family, empty state, settings-form pattern, daily-P&L-digest Block Kit shape, dedup ephemeral, quiet-hours bypass categories.
- `src/gekko/dashboard/templates/base.html.j2` — base template P5 EXTENDS, does not replace.
- `src/gekko/dashboard/static/tailwind.css` — stylesheet P5 APPENDS to, below the Phase 3 block (never modify existing rules).

**Single-operator, localhost-only, desktop scope** (D-18, P3 D-57): no phone/tablet breakpoints, no multi-user surfaces (those are Phase 6).

## Spacing Scale

Inherits the Phase 1→4 4-multiple grid verbatim. No new base tokens.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline icon-to-text gap, chip border-radius |
| sm | 8px | Compact element spacing (gap between trust badge and capital chip, button gaps inside modals) |
| md | 16px | Default element spacing (card padding, fieldset gaps, gap between portfolio-cap rows) |
| lg | 24px | Section padding (modal body, page padding, strategy-row outer padding) |
| xl | 32px | Layout gaps (between settings fieldsets, between page hero and content) |
| 2xl | 48px | Major section breaks (between strategies list and portfolio-caps section, empty-state padding) |
| 3xl | 64px | Page-level spacing (top margin on confirmation pages to give weight to the decision) |

**Policy (carry-forward):** outer layout (section gaps, container margins) uses 4-multiples only. Sub-component padding may use non-4 values where WCAG tap-target floors or typographic centering require — but the resulting outer dimension must land on a 4-multiple.

**Phase 5 sub-component exceptions (named, bounded):**

- **Promote / Demote / Scale-capital buttons + the typed-name confirm input:** minimum 44×44px tap target (WCAG 2.5.5 — Phase 2 lock). Achieved via 14px text + 12px vertical / 16px horizontal padding. Reuses the existing `.proposal-card-actions > button` floor.
- **Progress bar for the 10-approval streak meter:** 8px tall track (`.trust-progress-track`) — a deliberate non-4 inner height is NOT needed; 8px is on-grid.
- **Trust badge chip:** reuses the existing `2px 8px` chip padding (P2/P3 chip family) — visual parity with `chip-live` / `chip-expired`.

Inherits the Phase 1→4 role/size/weight matrix. P5 declares **no new roles** — every P5 surface maps onto an existing role.

| Role | Size | Weight | Line Height | Used By (P5) |

| Body-small / status (P3) | 14px | 400 (regular) | 1.4 | Streak progress caption ("7 of 10 clean approvals"), audit-line copy, auto-exec DM body |

**Weight policy (carry-forward, load-bearing):**

- **Safety-reserved weight 700 (bold):** reserved EXCLUSIVELY for the live-mode banner header and kill-active banner header (Phase 2 lock). **P5 introduces NO new 700-weight text.** The autonomy/anomaly surfaces signal severity via *color* (caution-yellow, live-red) and *copy*, not by stealing the reserved bold weight. Inline `<strong>` may bold a single load-bearing word inside body copy (e.g. "**cannot** be undone without a fresh confirmation"), matching the P3 quiet-hours-warning precedent.

**Typeface:** system font stack only — no Google Fonts, no `@font-face` (CSP + no-external-CDN).

## Surfaces & Interaction Contracts

Six surfaces in scope. Each declares placement, copy, state transitions, idempotency posture, ARIA semantics, and (where applicable) the Slack parallel. Several surfaces extend the existing strategies-list page and Settings page rather than introducing wholly new pages.

> **Standing rule across all P5 surfaces:** the UI is an *affordance*, never the *authority*. Trust level, capital ceiling, and the promotion gate are enforced server-side in the promotion helpers + OrderGuard. A forged POST (e.g. promote without a met streak) is rejected by the server gate (D-T18b) and by the safety-invariant AST/test gate ("auto-execute impossible unless criteria met" — CONTEXT Claude's Discretion). The UI merely renders state and collects intent.

### Surface 1: Per-Strategy Trust Level — Badge + Promote/Demote (TRUST-01, SC-1)

**Placement:** extends `strategies_list.html.j2` — each strategy renders as a `.strategy-row`. The trust badge sits in `.strategy-row-meta` alongside the existing `[LIVE]`/`[PAPER]` indicator and the (new) capital chip; promote/demote controls sit in `.strategy-row-actions`.

**Two orthogonal badges shown together** (CONTEXT "two orthogonal ladders, kept distinct"):

- paper/live axis (Phase 2, inherited): `[PAPER]` or `[LIVE]` chip.
- autonomy axis (P5): `PROPOSE-ONLY` (gray `.badge-trust-propose`) or `AUTO ✓` (green `.badge-trust-auto`).

**Row markup (per strategy):**

```html
<div class="strategy-row" data-strategy="{{ s.name }}">
  <div class="strategy-row-meta">
    <a href="/strategies/{{ s.name }}" class="font-semibold">{{ s.name }}</a>
    {% if s.account_mode == "LIVE" %}<span class="chip-live">LIVE</span>{% else %}<span class="badge-trust-propose">PAPER</span>{% endif %}
    {% if s.trust_level == "auto-within-caps" %}
      <span class="badge-trust-auto" aria-label="autonomy: auto-execute within caps">AUTO ✓</span>
    {% else %}
      <span class="badge-trust-propose" aria-label="autonomy: propose-only, human approval required">PROPOSE-ONLY</span>
    {% endif %}
    <span class="chip-capital" aria-label="capital ceiling">💵 ${{ s.capital_ceiling_display }}</span>
  </div>
  <div class="strategy-row-actions">
    {% if s.trust_level == "auto-within-caps" %}
      <button class="btn-primary"
              hx-post="/strategies/{{ s.name }}/demote-auto"
              hx-target="closest .strategy-row" hx-swap="outerHTML"
              hx-disable-elt="this"
              aria-label="Demote {{ s.name }} to propose-only. Takes effect next decision cycle.">
        Demote to propose-only
      </button>
    {% elif s.promotion_eligible %}
      <button class="btn-caution"
              hx-get="/strategies/{{ s.name }}/promote-auto/confirm-modal"
              hx-target="#modal-mount" hx-swap="innerHTML"
              aria-label="Promote {{ s.name }} to auto-execute within caps. Opens confirmation.">
        Promote to auto-execute
      </button>
    {% else %}
      <button class="btn-disabled"
              hx-get="/strategies/{{ s.name }}/promote-auto/blocked-modal"
              hx-target="#modal-mount" hx-swap="innerHTML"
              aria-label="Why can't this strategy be promoted yet?">
        Promote to auto-execute
      </button>
    {% endif %}
    <a class="btn-secondary" href="/strategies/{{ s.name }}/capital">Scale capital…</a>
  </div>
</div>
```

> Note: the "blocked" promote button is rendered (not hidden) but styled `.btn-disabled` and routes to the explanation modal — clicking it explains *why*, never silently does nothing (SC-5).

**Demote behavior (D-T04):**

- One click → `POST /strategies/{name}/demote-auto`. **No confirmation modal** — demotion is always-safe (removes autonomy, returns to HITL). Friction here would discourage the safe action.
- Server flips `trust_level → propose-only`, writes a `trust_demoted` audit event, returns the updated `.strategy-row` partial → HTMX swaps in place.
- **Takes effect on the next decision cycle** (SC-1): any in-flight cycle already past the auto-branch is not retroactively converted. The row immediately shows `PROPOSE-ONLY`; copy in the response confirms timing (see Copywriting).

**Promote behavior (eligible path, D-T04):**

- Click → loads the promote-confirm modal (Surface 2) into `#modal-mount`. Promotion is a deliberate sit-down decision — modal + typed-name confirm, no one-click.

**State transitions:**

- `propose-only` + eligible → click Promote → confirm modal → on confirm `POST .../promote-auto` → `auto-within-caps` (badge flips green).
- `auto-within-caps` → click Demote → `propose-only` (immediate badge flip; effective next cycle).
- `propose-only` + NOT eligible → click → blocked-explanation modal (Surface 5); no state change.

**Idempotency posture:**

- Promote/demote POSTs are state-machine guarded server-side: a second promote on an already-`auto` strategy no-ops and returns the current row (mirrors P3 cross-surface stale-card handling). No dedup table needed (these are deliberate operator actions on the authenticated dashboard, not at-least-once Slack deliveries).
- `hx-disable-elt="this"` bounds double-click on demote.

**ARIA / accessibility:**

- Badges carry `aria-label` spelling out the meaning (the visual `✓` / color is not the only signal).
- Disabled-promote button keeps `aria-label="Why can't this strategy be promoted yet?"` so it is discoverable as actionable-for-explanation, not dead.
- The updated row returned after promote/demote is wrapped so the badge change is announced (`role="status"` on the confirmation line).

**Slack parallel:** **none.** Promote/demote are dashboard + CLI only (D-T04). There is intentionally no Slack promote command.

### Surface 2: Promote-to-Auto Confirmation Modal (TRUST-01, SC-1)

**Route:** `GET /strategies/{name}/promote-auto/confirm-modal` → returns the modal partial into `#modal-mount`. Only reachable when `promotion_eligible` is true (the server re-checks; the blocked path renders Surface 5 instead).

**Modal layout** (reuses `.modal-backdrop` / `.modal` / `.modal-headline` / `.modal-body` / `.modal-actions` from the P2 modal family; typed-name confirm from the P2 promote-to-live precedent):

```html
<div class="modal-backdrop">

            hx-target="#promote-auto-result" hx-swap="outerHTML" class="mt-2">

               class="w-full mb-3" aria-label="Strategy name confirmation" placeholder="{{ name }}">
        <div class="modal-actions">
          <a href="#" hx-get="/modal/close" hx-target="#modal-mount" hx-swap="innerHTML">Cancel</a>
          <button type="submit" class="btn-caution">Promote to auto-execute</button>
        </div>
      </form>
      <div id="promote-auto-result"></div>
    </div>
  </div>
</div>
```

**Copy (deterministic):**

- Headline: `Promote to auto-execute`
- Caution block: `🚦 After this, {name} will place trades without asking you first, within its caps. Every auto-trade is still recorded and sent to you as an FYI — but there is no approve step. You can demote back to propose-only at any time, one click.`
- Qualification line: `This strategy qualified: {N} clean approvals, zero cap breaches in the window.`
- LIVE warning (live strategies only): `⚠️ This is a LIVE strategy. Auto-execute on live money still places real orders without a per-trade approval. (Your first live trade gate already passed.)`
- Confirm prompt: `Type the strategy name {name} to confirm:`
- Primary CTA: `Promote to auto-execute`
- Cancel: `Cancel`

**State transitions:**

- Submit with matching name → server re-verifies eligibility (defense-in-depth, D-T18b) AND name match → flips `trust_level → auto-within-caps` → writes `trust_promoted` audit event (with streak count + mode + capital ceiling in payload) → returns a success fragment that ALSO triggers an out-of-band swap of the strategy row (HTMX `hx-swap-oob`) so the badge flips green and the modal closes.
- Name mismatch → server returns the form re-rendered with an inline `.login-error`-style block: `That didn't match. Type {name} exactly to confirm.` No state change.
- Eligibility lost between modal open and submit (e.g. a cap breach landed) → server returns the blocked-explanation block (Surface 5) inside the modal; no promotion.

**Idempotency posture:** double-submit re-verifies; if already `auto`, returns the current row + a "already promoted" note. State-machine is the authority.

**ARIA / accessibility:** `role="dialog"` + `aria-modal="true"` + `aria-labelledby`. The LIVE warning is `role="alert"`. Confirm input has an explicit `aria-label`. Cancel is keyboard-reachable; focus lands on the text input on open.

**Slack parallel:** none.

### Surface 3: Capital Scaling Rung (TRUST-03, SC-3)

**Route:** `GET /strategies/{name}/capital` → a dedicated page (not a modal — it's a deliberate, separate ladder, D-T17) showing the current ceiling and a single numeric input. `POST /strategies/{name}/capital` applies the change.

**Recommended conservative defaults** (CONTEXT left exact numbers to planner/researcher discretion; these are the spec's prescriptive recommendations for a personal-use, small-dollars, swing-horizon tool starting human-in-the-loop):

| Cap | Default | Rationale |
|-----|---------|-----------|
| Max total exposure (% equity) | **50%** | Half the account can be deployed across all strategies at once; keeps half in cash as a runaway-loop buffer. |

| Max same-ticker exposure across strategies (% equity) | **15%** | Tighter than the 20% per-strategy `max_position_pct` ceiling (P1 schema) because it sums across strategies — prevents two strategies independently piling into the same name. |
| Max total daily loss (USD, portfolio-wide) | **$200** | A conservative absolute circuit breaker for a "start with small dollars" tool; stacks above per-strategy `max_daily_loss_usd`. Operator raises it deliberately as trust grows. |

> These defaults are **runtime-editable** (D-T09). They are written as `server_default` on the new `User` columns (mirrors the Phase-4 cost-ceiling default pattern) so a fresh install is conservative-by-default, never wide-open.
